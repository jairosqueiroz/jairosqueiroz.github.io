Azure VM Scheduler Case Study

Solving daily unplanned VM shutdowns by switching to intentional start/stop scheduling with Azure Functions timer triggers and managed identity.

Published: May 2026 Version: v1.0 Reading time: 10 min Region: East US / Brazil South
Azure VM Azure Functions Cost Optimization Automation Python
TL;DR

A Linux VM was shutting down daily at 03:48 UTC due to platform-level Hyper-V shutdown requests. The fix was to implement deterministic scheduling with Azure Functions: start at 06:00 UTC and stop at 22:00 UTC. This delivered predictable operations and lower costs versus Automation Runbooks and Logic Apps.

1. Problem Statement

The VM target-workload-vm was turning off unexpectedly every day despite auto-shutdown being disabled.

  • VM: Ubuntu 22.04 LTS
  • Size: Standard_D2s_v3
  • Pattern: shutdown around 03:48 AM UTC
  • No local cron or admin-issued shutdown commands

Root Cause Evidence

kernel: hv_utils: Shutdown request received - graceful shutdown initiated
systemd[1]: Reached target shutdown.target - System Shutdown.
systemd[1]: Finished systemd-poweroff.service - System Power Off.

Interpretation: graceful shutdown signals came from the virtualization layer, not from workload-level automation.

2. Architecture

Timer Trigger (06:00 UTC)  -> begin_start()     -> Azure VM
Timer Trigger (22:00 UTC)  -> begin_power_off() -> Azure VM
Authentication             -> Managed Identity  -> RBAC (VM Contributor)

3. Implementation Highlights

  • Python Azure Functions v4 using decorator model in function_app.py
  • Credential: DefaultAzureCredential() with system-assigned managed identity
  • Compute SDK: azure-mgmt-compute
  • Schedules: 0 0 6 * * * and 0 0 22 * * * (UTC)

Core Dependencies

azure-functions==1.18.0
azure-identity==1.14.0
    azure-mgmt-compute==30.6.0

5. Deployment Runbook

  1. Create or use a dedicated resource group (example: rg-automation-prod).
  2. Create a storage account for Functions runtime and package artifacts.
  3. Create a Python 3.11 Function App (Flex Consumption or Consumption plan).
  4. Enable system-assigned managed identity on the Function App.
  5. Grant least-privilege RBAC:
    • Virtual Machine Contributor on the target VM scope (or resource group scope when needed)
    • Storage Blob Data Contributor, Storage Queue Data Contributor, and Storage Table Data Contributor on the storage account
  6. Set deployment storage authentication to managed identity (no key-based auth).
  7. Set app settings using generic target values (example: AZURE_SUBSCRIPTION_ID, RESOURCE_GROUP=rg-workloads-prod, VM_NAME=target-workload-vm).
  8. Publish with func azure functionapp publish <app-name> --build remote and confirm triggers are registered.

6. Operations and Validation

  • Verify functions exist with az functionapp function list
  • Stream logs with az functionapp log tail
  • Confirm VM state transitions at schedule times
  • Track failures and retries in Application Insights

7. Lessons Learned

  • Platform maintenance events can be deterministic but opaque from inside the guest OS.
  • Intentional scheduling beats reactive troubleshooting for non-production windows.
  • Serverless automation can keep control-plane costs near zero for simple runbooks.
