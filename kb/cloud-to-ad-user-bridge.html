Cloud-to-AD User Bridge

Importing Microsoft Entra ID Users into On-Premises Active Directory — soft-match & hard-match strategies, idempotent PowerShell, production-ready runbook
v1.0 Published May 2026 Hybrid Identity Series 14 min read

TL;DR

Microsoft Entra Connect synchronizes AD → Entra ID natively, but does not write users from the cloud back into the on-premises directory. When you start cloud-first and later need users in AD (for hybrid Entra Kerberos, file shares, legacy apps), you need a controlled bridge: export cloud users, transform to AD attributes, create or update AD objects with consistent UPN / mail / proxyAddresses, then run a delta sync. Done correctly this triggers a soft match; done sloppily it produces duplicate identities.

1. Why this scenario exists

Two patterns produce a population of cloud-only users that the business later needs to access on-premises resources:

  • Cloud-first onboarding. A new tenant is provisioned, users are created directly in Entra ID, and only later does the team integrate an existing on-premises AD.
  • Legacy resource surfaces. A file server, a Kerberos-bound application, or an Azure Files share using AD DS authentication suddenly needs cloud-only users to authenticate. With Microsoft Entra Kerberos for Azure Files, hybrid identities are GA — cloud-only identities are still in preview as of May 2026.

In both cases you need cloud users to exist as hybrid identities: present in AD, synchronized to Entra ID, with both objects linked by the same ImmutableId.

2. How Entra Connect actually flows

Before fixing the gap, you need to be precise about which direction the synchronization service is willing to move data in.

ScenarioDirectionDefault Behavior
User created in on-prem ADAD → Entra IDSynced Becomes a hybrid identity
User created directly in Entra IDEntra ID → ADNot synced Remains cloud-only forever
AD password changedAD → Entra IDReplicated Via Password Hash Sync
Self-service password reset in cloudEntra ID → ADOpt-in Requires Password Writeback
M365 group created in cloudEntra ID → ADOpt-in Requires Group Writeback
Device registered as Entra-joinedEntra ID → ADOpt-in Requires Device Writeback
There is no "User Writeback" feature for cloud-only users
Several writeback features exist (password, group, device) but none of them cover the case of a user that was born in the cloud. That has to be done by you.

4. Soft match vs. hard match

The matching engine in Entra Connect tries to recognize that "this new AD user" and "this existing Entra user" are the same person. There are two mechanisms.

Soft match — attribute-based

Entra Connect matches on primary SMTP / mail attribute (case-insensitive). If mail in AD equals mail in Entra, the objects are linked automatically. UPN is used as a secondary signal but mail is the strongest deterministic key.

When to use: default. Always try soft match first — it requires no special tooling and is fully reversible.

Hard match — ImmutableId

You explicitly set the cloud user's ImmutableId property to a base64-encoded value derived from the AD object's objectGUID. Entra Connect will then bind these two objects together regardless of mail/UPN.

When to use: when soft match fails, when mail attributes diverge, or when you need deterministic linkage for audit purposes. Hard match requires PowerShell on the cloud side and is harder to undo.

The duplication risk
If you create an AD user with a mismatched mail/UPN and Entra Connect picks it up before you fix it, you end up with two objects in Entra (the original cloud-only and a new synced one). Cleaning this up is painful — always validate attributes before running delta sync.

5. Runbook — step by step

The process below assumes you have a tenant where some users were created cloud-only, an on-prem AD with Entra Connect installed, and a target OU + group structure already designed.

Export cloud-only users from Entra ID

Pull every user where onPremisesSyncEnabled is null/false. Microsoft Graph PowerShell is the cleanest path.

PowerShellexport-cloud-users.ps1
# Connect with the User.Read.All scope
Connect-MgGraph -Scopes "User.Read.All"

# Export every user that is NOT on-premises synced
Get-MgUser -All -Property "Id,DisplayName,UserPrincipalName,Mail,UserType,OnPremisesSyncEnabled,Identities,CompanyName,CreationType" |
    Where-Object { $_.OnPremisesSyncEnabled -ne $true } |
    Select-Object Id, DisplayName, UserPrincipalName, UserType, OnPremisesSyncEnabled,
        @{N="Identities";E={$_.Identities | ConvertTo-Json -Compress}}, CompanyName, CreationType |
    Export-Csv -Path ".\exportUsers_$(Get-Date -Format 'yyyy-M-d').csv" -NoTypeInformation -Encoding UTF8

Transform the CSV to AD-ready format

Re-shape the export into the columns AD needs: UserPrincipalName with the on-prem domain suffix, sAMAccountName, GivenName, Surname, Department, plus CloudObjectId for the audit trail.

The UPN suffix change (from tenant.onmicrosoft.com to your verified custom domain like corp.local) is the most important transformation. The verified domain must be added as an Entra ID custom domain before the sync runs.

Pre-flight match report — detect duplicates before writing

Run an idempotent dry-run that classifies each row as create, update, skip, or conflict. Never skip this step.

PowerShellmatch-report.ps1
# Read the AD-import CSV produced in step 2
$rows = Import-Csv ".\exportUsers_2026-5-4-for-ad-import.csv"

$report = foreach ($r in $rows) {
    $ad = Get-ADUser -Filter "sAMAccountName -eq '$($r.sAMAccountName)'" -Properties mail, userPrincipalName, memberOf -ErrorAction SilentlyContinue
    [PSCustomObject]@{
        CloudUPN       = $r.CloudUPN
        LocalUPN       = $r.UserPrincipalName
        sAMAccountName = $r.sAMAccountName
        ExistsInAD     = [bool]$ad
        Department     = $r.Department
        Groups         = "GG-$($r.Department)-FS"
        Dn             = $ad.DistinguishedName
    }
}
$report | Export-Csv ".\exportUsers_2026-5-4-match-report.csv" -NoTypeInformation

Create or update AD users (idempotent)

The script must be safe to run multiple times. Use Set-ADUser for existing objects and New-ADUser only when the dry-run flagged the row as create.

PowerShellprovision-ad-users.ps1
$rows   = Import-Csv ".\exportUsers_2026-5-4-for-ad-import.csv"
$ouPath  = "OU=CloudBridged,OU=Users,DC=corp,DC=local"
$tempPwd = ConvertTo-SecureString "TempP@ssw0rd!ChangeMe" -AsPlainText -Force

foreach ($r in $rows) {
    $existing = Get-ADUser -Filter "sAMAccountName -eq '$($r.sAMAccountName)'" -ErrorAction SilentlyContinue

    if (-not $existing) {
        New-ADUser `
            -Name              $r.DisplayName `
            -DisplayName       $r.DisplayName `
            -GivenName         $r.GivenName `
            -Surname           $r.Surname `
            -SamAccountName    $r.sAMAccountName `
            -UserPrincipalName $r.UserPrincipalName `
            -EmailAddress      $r.UserPrincipalName `
            -Department        $r.Department `
            -Path              $ouPath `
            -AccountPassword   $tempPwd `
            -Enabled           $true `
            -ChangePasswordAtLogon $true
        Write-Host "Created $($r.UserPrincipalName)" -ForegroundColor Green
    } else {
        Set-ADUser -Identity $existing `
            -UserPrincipalName $r.UserPrincipalName `
            -EmailAddress      $r.UserPrincipalName `
            -Department        $r.Department
        Write-Host "Updated $($r.UserPrincipalName)" -ForegroundColor Yellow
    }
}

Add users to AD groups by department

File-share permissions, Conditional Access groups, and Entra Kerberos RBAC roles all hang off AD groups. Map each user to the group(s) that match their department.

PowerShelladd-to-groups.ps1
$groupMap = @{
    "HR"    = "GG-HR-FS"
    "IT"    = "GG-IT-FS"
    "Sales" = "GG-Sales-FS"
}

foreach ($r in $rows) {
    $group = $groupMap[$r.Department]
    if ($group) {
        Add-ADGroupMember -Identity $group -Members $r.sAMAccountName -ErrorAction SilentlyContinue
    }
}

Trigger an Entra Connect delta sync

Run on the Entra Connect server. Delta is fast (seconds to a minute).

PowerShelltrigger-sync.ps1
Import-Module ADSync
Start-ADSyncSyncCycle -PolicyType Delta

Validate the match in Entra ID

Each previously cloud-only user should now show onPremisesSyncEnabled = True and have an onPremisesImmutableId populated.

PowerShellvalidate-match.ps1
Connect-MgGraph -Scopes "User.Read.All"

$rows = Import-Csv ".\exportUsers_2026-5-4-for-ad-import.csv"
foreach ($r in $rows) {
    $u = Get-MgUser -UserId $r.CloudObjectId -Property OnPremisesSyncEnabled,OnPremisesImmutableId,UserPrincipalName
    [PSCustomObject]@{
        UPN     = $u.UserPrincipalName
        Synced  = $u.OnPremisesSyncEnabled
        Anchor  = $u.OnPremisesImmutableId
    }
}
Success looks like this
Every row in the validation output shows Synced = True and a non-empty Anchor. The cloud user and the AD user are now bound — soft match worked.

Fallback — hard match by ImmutableId (only if soft match fails)

If a row failed to match (typically because mail attributes diverge), set the ImmutableId explicitly. Convert the AD objectGUID to base64 first.

PowerShellhard-match.ps1
$adUser      = Get-ADUser "ana.costa" -Properties objectGUID
$immutable   = [Convert]::ToBase64String($adUser.objectGUID.ToByteArray())
$cloudUserId = "d0c821f6-4609-42c0-aadf-9571dd810da9"

Update-MgUser -UserId $cloudUserId -OnPremisesImmutableId $immutable

6. Going from lab to production

The lab version above is fine for a one-shot bootstrap. To run this as an ongoing operation, you need three additional properties.

6.1 An explicit match policy

Document the rules in a single page that ops can follow without reading scripts:

  • Default: soft match by mail. UPN suffix must use a verified Entra ID custom domain.
  • Fallback: hard match by ImmutableId only when an attribute conflict has been investigated and approved.
  • Forbidden: creating an AD user with an unverified UPN suffix. This produces orphaned objects and breaks federation.

6.2 An idempotent script with a duplication report

Every run produces three artifacts:

  • match-report.csv — what would happen, written before any change.
  • execution-log.csv — what actually happened, written after every operation.
  • conflicts.csv — rows that were skipped because of attribute conflicts. Reviewed by ops before the next run.

6.3 Scheduled execution with manual approval gate

A daily scheduled task on the Entra Connect server (or a dedicated jump box) runs the export and produces match-report.csv. The actual provisioning script runs only after a human approves the report — typically by moving the file to a ./approved/ folder. This prevents drift and surprise creates.

Treat the bridge as a temporary control
If the long-term direction is "cloud-first identity", the bridge is a transitional pattern, not a destination. The endgame is removing the AD dependency entirely (e.g., Microsoft Entra Kerberos with cloud-only identities once GA).

SymptomLikely CauseFix
Duplicate user appears in Entra ID after syncUPN/mail mismatch, soft match couldn't bindStop sync, soft-delete the duplicate, fix attributes, hard match by ImmutableId, resync
onPremisesSyncEnabled stays nullAD object is in an OU not included in Entra Connect scopeMove the user to a synced OU, then run delta sync
